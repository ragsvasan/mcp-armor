# mcp-armor — Three-Lens Security Audit

**Date:** 2026-06-12
**Scope:** `mcp-armor` server-side SDK (defends MCP servers against CoSAI T1–T12).
**Method:** (A) live attack rig — same inner MCP app stood up UNARMORED vs ARMORED, attacked
per-threat over the real `ArmorMiddleware` ASGI path; (B) enterprise-fit static review; (C)
defense-honesty / bypass trace from adapter → guard → engine. Live rig:
`/tmp/armor_rig.py`, raw results `/tmp/armor_rig_results.json`.

---

## Executive Summary

**Does enabling armor actually defend? — Mostly YES, and provably so.** Unlike the companion
scanner (whose flagship check was defined, tested, and never called), mcp-armor's *runtime*
enforcement is largely real. Every engine I could isolate **blocked its attack live while the
unarmored baseline succeeded**: T1 (missing token, `alg=none`), T3 (cmd/path/SQL injection), T4
(prompt injection in args + response), T5 (SSN leak — unarmored returns `123-45-6789`, armored
returns `-32004`), T8 (SSRF to `169.254.169.254` and RFC1918), T9 (response injection), T10
(call budget → HTTP 429). The architecture is fail-closed by default (`dry_run=False`, auth
errors never suppressed even in dry_run, unexpected engine exceptions → `-32603` block), the
ASGI adapter buffers the full upstream response and runs response engines *before* anything
reaches the client, and RE2 patterns are compiled once at startup, not per request.

**But the headline finding is the same precedent class you asked me to hunt — and it's live-confirmed.**
The T3 JSON-schema validator (`register_tool_schemas` → `ValidationEngine.register_tools`) is
**defined, unit-tested, and never called by any adapter.** With the default `strict_schema=True`,
`_tool_schemas` is always empty on the live path, so **every `tools/call` is rejected with "no
registered schema."** I proved it: a benign `echo` call through the armored middleware returns
`-32602 Validation error`. So the shipped default either (a) blocks 100% of legitimate tool calls
(self-DoS), or (b) the operator disables `strict_schema` to function — at which point the
advertised T3-005 schema enforcement silently never runs. The claimed defense is non-functional
on every real path.

**Three more "claimed-but-not-delivered" gaps:** T7 DPoP/session binding (`bind_session_to_dpop:
true`) is a label nothing reads; T8's startup `0.0.0.0` bind check is dead on the config path; and
the T12 audit chain has a rollback-to-empty bypass (delete the log → no verification on next boot).
None of these are fail-*open* in the dangerous sense for the request path, but each is a matrix
claim the code doesn't honor.

**Would an enterprise embed it today? Not yet** — for provenance and packaging reasons more than
defense reasons. PyPI serves `mcp-armor 1.0.x`, `pyproject.toml` says `0.1.0`, the last commit
claims `v0.2.0`, and there are **no git tags** — you cannot trace the installed artifact to source.
The documented quickstart config (`cosai.yaml.example`) **cannot serve a single request** out of
the box (needs an undocumented `ARMOR_SESSION_SECRET`, a writable `/var/log/mcp-armor`, and a JWKS
+ `endpoint_uri` it never mentions). And there is **zero performance evidence** for a library whose
entire pitch is the per-request hot path.

## Confirmed Findings (deduped, prioritized)

### CRITICAL

**A1 | T3 | JSON-schema validation is dead on every live path (self-DoS by default).**
`mcp_armor/guard.py:232` defines `register_tool_schemas()`; the only callers are the definition
itself and `tests/`. No adapter ever calls it, and `ValidationEngine.on_response`
(`validation.py:209`) is a no-op, so `_tool_schemas` is never populated at runtime. With default
`strict_schema=True`, `validation.py:197-202` raises `ValidationError` "has no registered schema"
for **every** `tools/call`. **Live-proven:** benign `echo` → `-32602`.
*Recommendation:* auto-register schemas from the observed `tools/list` response inside the adapters
(mirror Integrity/SupplyChain `on_response`), or add `ValidationEngine.on_response` that registers
from the manifest. Add an integration test that drives `tools/list → tools/call` through
`ArmorMiddleware` and asserts a real `inputSchema` violation is blocked **and** a valid call passes.

### HIGH

**A2 | T7 | DPoP/session binding is a no-op label.**
`SessionEngine.__init__` stores `self._bind_to_dpop` (`session.py:44`) and config reads
`bind_session_to_dpop: true` (`config.py:377`), but the flag is **never referenced again** (grep:
only the assignment). The signer (`_session_token.py`) binds only `transport` into the MAC, never a
DPoP thumbprint or principal — the module docstring concedes the token is bearer-equivalent and
"replayable until the signing secret is rotated." The "bound to DPoP/session, rejected on reuse"
claim is false. *Recommendation:* either bind the DPoP JKT into the MAC and reject on mismatch (with
a regression test that a token minted under one key fails under another), or remove the flag and the
YAML key and document T7 as transport-bound continuity only.

**A3 | T8 | Startup bind-address (`0.0.0.0`) check is dead on the config path.**
`T8Config` (`config.py:208-210`) has no `bind_host`/`bind_port` field, and `_from_armor_config`
constructs `NetworkEngine(allow_public_bind=…, block_rfc1918_ssrf=…)` with no bind host. So
`network.py:116` `if self._bind_host is not None:` is always False — `check_bind_address()` never
runs. `cosai.yaml.example` advertises `allow_public_bind: false  # 0.0.0.0 → error at startup`,
which can never fire from a YAML-built guard. (Per-request SSRF, `on_request`, **is** correctly
wired and fail-closed — proven live.) *Recommendation:* add `bind_host`/`bind_port` to `T8Config`
and the schema, thread them into the constructor, and test that `bind_host: 0.0.0.0` raises at
`guard.startup()`.

**A4 | T12 | Audit chain has a rollback-to-empty bypass.**
Chain verification only runs when the log already exists: `audit.py:291`
`if self._verify_on_startup and self._path.exists(): … else: await self._seed_state()`. An attacker
with write access who deletes the log (plus the `.hwm` sidecar and `.hmac_enabled` marker) makes
`path.exists()` False, so startup seeds from empty with **no verification** — the HWM-absent guard
(`audit.py:203`) lives inside `_sync_verify_chain`, which is skipped. Full-history erasure is
undetectable on the next boot. (When the file *is* present, the chain is genuinely tamper-evident:
per-record hash over `seq`, `prev_hash` linkage, gap detection, optional sticky HMAC.)
*Recommendation:* persist the HWM/marker in a store independent of the log; if the marker exists but
the log is absent, raise `AuditChainError` instead of seeding empty. Regression test: write N
records, delete the log (keep sidecar), assert startup raises.

**A5 | enterprise | Version/provenance is unresolvable.**
`pyproject.toml:7` = `0.1.0`; last commit message = `v0.2.0`; **PyPI serves `1.0.0`–`1.0.2`**; `git
tag -l` is empty while `publish.yml` triggers only on `v*` tags. The installed artifact maps to no
tagged commit. An enterprise cannot pin a security dependency it cannot trace to source.
*Recommendation:* reconcile the version, tag releases, map CHANGELOG entries to commits. (Positive:
`publish.yml` uses PyPI OIDC trusted publishing + Sigstore provenance.)

**A6 | enterprise | README "by default" claims contradict shipped defaults.**
README:108 claims Ed25519 registry signatures are "enforced … by default," but
`require_registry_signature` defaults **False** (`config.py:230`, `cosai.yaml.example:95`) and
`registry_public_key` defaults None. `ARMOR_AUDIT_HMAC_KEY` is unset by default (`audit.py:46`), so a
first-time deploy gets a tamper-*evident-looking* but forgeable audit log. The example config
produces a server that **looks** protected with signing + tamper-proof audit OFF. *Recommendation:*
either flip the defaults to match the docs, or fix README/example to state these are opt-in; make
`ARMOR_AUDIT_HMAC_KEY` mandatory at startup when T12 is enabled outside a dev profile.

### MEDIUM

**B1 | T4/T5/T9 | Response engines are block-only; docstrings claim "scrub/sanitize/strip."**
`ProtectionEngine` ("Scrubs PII…"), `TrustEngine` ("Sanitizes… returns safe text", config
`strip_injection_patterns`) only **raise** in their wired `on_response`; nothing is stripped from
the wire. The HTTP adapter sends the original buffered `resp_raw` bytes regardless
(`fastapi.py:447`); the decorator path returns the original `result` (`guard.py:498`); the
dispatcher returns `raw_resp` (`dispatcher.py:47`). This is fail-*closed* (safer than scrubbing) but
the naming misrepresents behavior — there is no redaction, only all-or-nothing blocking.
*Recommendation:* rename to reflect block semantics, or implement actual mutation and re-serialize
the post-response object.

**B2 | enterprise | Documented quickstart config cannot serve a request.**
`CoSAIGuard.from_config("cosai.yaml.example")` fails in sequence on: missing `ARMOR_SESSION_SECRET`
(RuntimeError, refuses to start); then `PermissionError: /var/log/mcp-armor` (default T12 path); and
T1 sets `require_dpop: true` with no `jwks`/`endpoint_uri` (`auth.py:411-420` would raise). All
fail-closed (good direction) but none are discoverable from the YAML. *Recommendation:* ship a
runnable example (writable default audit path, a sample JWKS/endpoint_uri, a generated session
secret) and document the required env vars at the top of `cosai.yaml.example`.

**B3 | T6/T10 | dispatcher adapter mints a fresh session per call.**
`dispatcher.py:36` calls `mint_session_id` on every invocation, so per-session state (T6 manifest
baseline, T10 call/wall-clock budget) never accumulates on that transport. Combined with B1's
discarded `_run_response` return, the dispatcher provides no cross-call integrity or budget
coverage. *Recommendation:* document the dispatcher as single-call-session only; do not claim T6/T10
coverage there.

**B4 | T6 | Drift baseline is in-process; multi-worker is acknowledged fail-open.**
`fastapi.py:283-292` documents that a worker with no stored context does `CoSAIContext.new()` which
resets `tool_manifest_hash` — an attacker can route the second `tools/list` to a different worker to
escape drift detection. Enforced only within single-worker session continuity. *Recommendation:*
back the session/manifest baseline with a shared store before claiming T6-001 in horizontally-scaled
deployments, or refuse to start with `workers > 1` until then.

**B5 | enterprise | No performance evidence for a hot-path library.**
Zero benchmarks in the repo (grep `benchmark/timeit/perf_counter` = 0). The 12-engine chain runs
sequentially per request (`guard.py:274`), Boundary scans 3 surfaces (original + normalized +
base64-decoded) per call. *Recommendation:* add a `pytest-benchmark` harness, publish p50/p99 chain
overhead before claiming enterprise hot-path readiness. (Positive: RE2 compiled once at init, not
per request.)

**B6 | enterprise | `dry_run` has no hard guard against prod.**
`dry_run` disables all non-auth enforcement; it defaults False and warns at load + construction
(`config.py:447`, `guard.py:63`), but nothing prevents `dry_run: true` shipping to prod beyond a log
line. *Recommendation:* env-gate it (refuse unless `ARMOR_ALLOW_DRY_RUN=1`) or emit at startup-banner
ERROR level.

**B7 | enterprise | Destructive-tool gate ineffective for autonomous agents.**
By the project's own admission (`SECURITY.md:79-91`), the T2 human-in-the-loop confirm-token gate
"provides no protection regardless of `echo_confirm_token`" for fully autonomous clients with no
human intercept. *Recommendation:* keep the honest docs, but treat this as not-yet-delivered for
autonomous deployments; pair with RFC 9470 step-up auth (roadmap).

**B8 | enterprise | TS sidecar is doc-only.**
`docs/TYPESCRIPT.md` ships a copy-paste `proxy_server.py` snippet — no sidecar module, entrypoint,
Dockerfile, or image exists; stdio TS servers are uncovered; the loopback hop is unbenchmarked.
*Recommendation:* ship `python -m mcp_armor.sidecar` + a published image, or label it a DIY recipe.

### LOW

**C1 | enterprise | Dependencies floor-pinned only** (`pyproject.toml:23-30`) — no upper bounds,
lockfile, or hashes. Ship a hash-pinned lockfile for reproducible installs.

**C2 | T1 | `_check_rsa_key_size` fail-open skip** (`auth.py:143-146`): `except Exception: return`
silently skips the NIST key-size floor if modulus decode throws. Bounded but should reject, not skip.

**C3 | enterprise | Compliance/SIEM is documentation only.** `THREAT_MAPPING.md` is an informational
mapping (not certified controls); SIEM/SOAR emitter is roadmap (grep `siem/syslog/splunk` = 0). The
audit log is a portable JSONL a SIEM can tail; state plainly that export is file-tail-only today.

## 12-Engine Honesty Scorecard

|---|---|---|---|---|
| T1 | auth | JWT/DPoP, alg-confusion, jti replay | **YES — live-blocks** missing/`alg=none`; refuses boot w/o JWKS | fail-closed (C2 minor skip) |

| T6 | integrity | mid-session manifest drift | **PARTIAL** — real single-worker; **multi-worker fail-open (B4)** | fail-open across workers |
| T7 | session | session binding to **DPoP** | **NO — `bind_to_dpop` is a no-op label (A2)** | transport-bound only; replayable |
| T8 | network | SSRF + `0.0.0.0` startup bind block | **PARTIAL** — SSRF live-blocks; **startup bind check DEAD (A3)** | per-request fail-closed |
| T9 | trust | sanitize/`strip` LLM output | **YES — live-blocks** (raises, does not strip) | fail-closed; B1 misnamed |

## Enterprise-Readiness Gaps (ranked — would block adoption)

1. **Provenance & packaging (A5)** — version chaos + no tags; can't trace PyPI `1.0.x` to source.
2. **Docs/defaults mismatch (A6, A1)** — example config yields "looks-protected-but-isn't" for
   schema validation, registry signatures, and audit-HMAC; the marquee schema check self-DoSes.
3. **No performance evidence (B5)** + **TS story is doc-only (B8)** — a hot-path library with no
   benchmarks, and the entire TS-server market is a hand-assembled snippet.

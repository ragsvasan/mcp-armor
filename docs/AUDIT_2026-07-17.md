# Security audit — 2026-07-17

Three-layer review of mcp-armor and its reverse-proxy sidecar, followed by a T1
panel on the remediation. Remediated on branch `security/audit-2026-07-remediation`.

**Method**
1. Mechanical scanners — `pip-audit`, `bandit`, `semgrep` (236 rules), `trivy`.
2. Opus 4.8 core audit — auth/authz, audit, validation, egress, SSRF, remediation code.
3. Fable 5 deep-dive — sidecar reverse proxy, DPoP (RFC 9449 §4.3), MCP §3.2 gate, Ed25519.
4. Remediation panel — defense + adversary + `/security-review` + cryptographic-systems
   and MCP-protocol domain personas, on the fix diff.

## Result

- **3 HIGH, 6 MEDIUM** code findings + LOW/INFO, all fixed and regression-tested
  (`tests/security/`). Full suite: 678 passing.
- The auth/authz **core is sound** — no direct authentication/authorization bypass in
  token verification, default-deny, session HMAC, signatures, DPoP alg-confusion, or
  Ed25519 manifest verification. SAST (bandit + semgrep) and container/secret (trivy) clean.
- The three HIGHs shared **one root cause: *inspect ≠ forward/persist*** — the guard
  inspected a normalized/parsed/truncated/subset view while forwarding or persisting the
  raw/full thing. The remediation applies one rule everywhere: *inspect exactly the bytes
  or fields you forward or persist.*

## HIGH

| # | Location | Issue → fix |
|---|---|---|
| H1 | `adapters/fastapi.py`, `types.py` | Response scanning saw `str(dict)[:65536]` (non-JSON → `{}` scanned clean) while the full raw body egress'd → T5 PII/secret bypass. **Fix:** scan the full body; fail closed on non-object/unparseable bodies when a response engine or the `tools/list` scope filter is active. |
| H2 | `sidecar.py` | Client `Content-Length` not stripped before `httpx.stream(content=body)` → CL/TE desync smuggles an uninspected second request upstream. **Fix:** drop `content-length` from the forwarded headers; httpx recomputes it. |
| H3 | `engines/audit.py` | `chain_hash`/`chain_hmac` covered only fixed fields; persisted `extra` (e.g. `violation_class`/`violation_detail`) was unauthenticated and editable without breaking the chain. **Fix:** canonicalize the whole record minus the integrity fields. |

## MEDIUM

- **M1** `engines/audit.py` — the count-anchoring `.hwm` sidecar was unsigned, so trailing
  records could be truncated undetected even with the HMAC key set. **Fix:** HMAC the
  sidecar; treat a missing/mismatched MAC as tamper when a key is configured.
- **M2** `engines/validation.py` — `strict_schema` silently no-op'd if `jsonschema` was
  absent. **Fix:** fail closed at startup.
- **M3** `engines/network.py` — SSRF scan failed open on DNS-resolution failure and missed
  numeric-IP encodings. **Fix:** fail closed for unresolvable hosts when SSRF blocking is
  on; normalize decimal/octal/short IPv4 before resolving.
- **M4** `engines/auth.py` — DPoP htu binding was skipped when `endpoint_uri` was unset,
  allowing cross-endpoint proof replay. **Fix:** fail closed when `endpoint_uri is None`.
- **M5** `engines/auth.py` — the DPoP jti replay cache was per-process (multi-worker replay
  window). **Fix:** pluggable jti store (in-memory default, startup warning for multi-worker;
  no hard Redis dependency).
- **M6** `sidecar.py` — client `Authorization`/`Cookie`/`DPoP` forwarded verbatim to a
  potentially-compromised upstream. **Fix:** opt-in `strip_upstream_auth`.

Plus the HS\* hardening: HS256/384/512 access-token algs are dropped unless the JWKS is
exclusively `oct` keys, closing both RS/HS confusion and the joserfc empty-key surface.

## Remediation panel — follow-ups on the fixes

The T1 panel on the remediation itself found that several fixes were incomplete or
introduced new gaps (all fixed, tested in `tests/security/test_regression_panel_fixes.py`):

- **`types.py`** — a LOW fix coerced non-object JSON-RPC params to `{}`, reintroducing the
  scan/forward asymmetry on the *request* side (guard scans empty; raw array forwarded).
  **Fix:** reject non-object params (fail closed); the raw dispatcher's `from_dict` moved
  inside its try so it fails closed there too.
- **`adapters/fastapi.py`** — the non-object fail-closed only triggered for *content*
  scanners, so a `tools/list` scope filter was silently skipped (authz bypass). **Fix:**
  fail closed for `tools/list` too.
- **Response size cap** — reused the 64 KB *request* cap (rejecting legitimate tool
  results) and was checked *after* the full body was buffered (OOM). **Fix:** own 10 MiB
  ceiling, enforced incrementally during buffering; wired through the sidecar.
- **`engines/resources.py`** — the denial-of-wallet fix persisted `_session_budgets` but
  never bounded it; mint-and-abandon paths (HTTP open→zombie-reap, `wrap_dispatcher`) grew
  it without limit. **Fix:** LRU-bounded ledger.

Crypto persona: no findings. One LOW residual is documented, not code-fixed: the non-re2
regex-windowing fallback can split a single match span wider than the window overlap —
`google-re2` is a pinned core dependency, so this only manifests in a degraded install and
is pre-existing, not a regression.

## Dependencies

A bare `pip-audit` over the shared interpreter reported 59 CVEs across 22 packages, but a
resolution of mcp-armor's *declared* closure (base + all extras, 73 packages) pulls the
already-fixed versions of every flagged package — the rest (torch, onnx, chromadb,
langgraph, aiohttp, …) are not in mcp-armor's dependency set. The one real hardening:
`joserfc` floor raised to `>=1.6.8` (empty-key HMAC bypass PYSEC-2026-2528 + RFC7797
size bypass), matching the existing F8 alg-confusion intent. Recommendation: run
security scans against a resolved lockfile, not the shared dev environment.
